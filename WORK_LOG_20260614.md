### 3. 修复致命 XamlParseException 死循环

**问题背景**：
- 一点选物料（如门厅 + 瓷砖），应用立即崩溃 / 死循环，日志疯狂刷屏。
- 异常信息：
  ```
  System.Windows.Data Error: 5
  '无法对 SelectedMaterial 类型的只读属性 TotalPrice
   进行 TwoWay 或 OneWayToSource 绑定。'
  ```

**根因分析**：
- `SelectedMaterial.TotalPrice` 是**计算属性**（`=> UnitPrice * Quantity`），只有 getter、没有 setter。
- 但 `INotifyPropertyChanged` 的 `OnPropertyChanged(propertyName)` 通用方法在某些路径下会**手动触发 `TotalPrice` 通知**（不在 `Quantity` setter 内），WPF 看到属性有 PropertyChanged 就推断绑定方向为 **TwoWay**，回写时找不到 setter → 抛错。
- 接着 WPF Measure 过程因绑定错误反复重试 → 死循环。

**解决方案**（三处同时修）：
1. **`OnPropertyChanged` 方法内不再手动触发 `TotalPrice` 通知**（只让 Quantity setter 显式触发）。
2. **`Quantity` setter 内显式调用**：
   ```csharp
   OnPropertyChanged(nameof(TotalPrice));
   ```
3. **XAML 绑定保险**：`TotalPrice` / `UnitPrice` 都加 `Mode=OneWay`，明确告诉 WPF 这是单向绑定，不要尝试回写。

**涉及文件**：
- `FactoryProductManager/Views/AddProductMaterialWindow.xaml.cs`（`SelectedMaterial` 类的 `OnPropertyChanged` / `Quantity` setter）
- `FactoryProductManager/Views/AddProductMaterialWindow.xaml`（DataTemplate 内 `Mode=OneWay`）

### 4. 自适应高度优化（应用户反馈）

**用户反馈原文**：
> "上半部分能不能根据选中的部位数量完全现出来，这样就避免要去拉中间的分界框"

**问题**：
- 原来上半 `RowDefinition Height="*"` + 中间 `8` + 下半 `2*` + `GridSplitter`，固定 1:2 比例。
- 显示 5 个部位时上半刚好用完，但下半被强行分配 2/3 高度，**详情区底部留出一大块空白**。
- 用户还得拖动 GridSplitter 调比例，体验差。

**解决方案**：
- 主 Grid Row 改为 `Auto` + `*`：
  - `RowDefinition Height="Auto"`（上半：高度 = 内容实际高度）
  - `RowDefinition Height="*"`（下半：占剩余空间）
- **删除 GridSplitter**（用户明确说不需要拖动条）
- **删除上半 ScrollViewer**（让 `StackPanel` 自然撑开，不限制高度）
- 详情区 ScrollViewer 保留（物料多了只滚下半）

**新结构图示**：
```
┌──────────────────────────────────┐
│  标题栏                            │
├──────────────────────────────────┤
│  上半（Auto - 完全显示）           │
│  ┌──┬─────────────────────────┐   │
│  │N │  部位选择（全部显示）      │   │  ← 高度 = 内容高度
│  │a │                          │   │
│  │v │                          │   │
│  └──┴─────────────────────────┘   │
├──────────────────────────────────┤
│  下半（* - 占剩余空间）            │
│  ┌──────────────────────────┐    │
│  │ 已选物料详情（独立滚动）     │    │  ← 高度 = 剩余
│  │  [卡 1]                     │    │
│  │  [卡 2]                     │    │
│  └──────────────────────────┘    │
├──────────────────────────────────┤
│            [取消]  [确定]          │
└──────────────────────────────────┘
```

## 技术要点

- WPF 列表渲染：`ItemsControl` + `DataTemplate` 比 `StackPanel` 动态 `Add(UIElement)` 更高效、更易维护，且自动支持数据绑定。
- 计算属性的 PropertyChanged 通知必须在**依赖属性的 setter 内显式触发**，不能在 `OnPropertyChanged(string)` 通用方法里盲目触发，否则 WPF 推断为 TwoWay 绑定导致回写失败。
- Grid 行高：`Auto` = 内容实际所需高度；`*` = 剩余空间均分。两者搭配可实现"内容自适应 + 剩余占满"的弹性布局。
- 父子两个 ScrollViewer 会冲突：内层 ScrollViewer 限定高度后，外层就感知不到子元素真实高度。本场景上半部分**不要**加 ScrollViewer，让 StackPanel 自然撑开。
- GridSplitter 用法：可拖动分隔条改变 RowDefinition 比例，但会引入用户认知负担（要拖动才知道有这功能），适合高级用户场景，**不适合普通业务流程表单**。
